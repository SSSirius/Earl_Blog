3月工作杂记

Mar 23 2020

不知不觉开始社畜人的日子已经过了半年了，这半年时不时的又在记录一些工作上，专业上的事情，但是自己操作时候往往会忘记或者遗漏。趁此机会整理一下，与君共勉。

交互的基础步骤

这可能是一个老生常谈的方法论问题。

业务目标 -> 业务逻辑和优先级 -> 页面架构 -> 流程状态 -> 模块信息结构

目前我的实操中在流程状态阶段已经可以hold 住了，但是down到具体的模块信息架构，常常因为各种原因，把信息的全部都揉杂在一块。这也充分显示出自己处理问题的不细致（被骂了好几次哦

🤦‍♀️

那么应该怎么去拆各种信息层级呢？

  1. 把所有信息罗列出来，必要的和不必要的分类
  2. 相关联的信息做整合，挨在一起
  3. 重点信息突出，决定视觉层级，非重要信息简化，不不需要用过多颜色
  4. 结构提炼，基础信息概括各个层级
  5. 替换信息，按照之前的优先级突出不同层级结构，非必要信息弱化

这样基本的交互信息结构就完成了。
很多时候为了快速支持业务，快速出图，这一层都会被略过。但是这是最最基础的一个环节，应该在日常设计中潜移默化的提醒自己，去完成它。
更重要的是：
设计是不会有完成状态的，每当有机会去迭代之前的稿子时，就要重新审视一下之前的方案问题，是否能更好，更流畅。

商业化的考虑

还是回到最近比变化了的业务上来。原来做业务设计，更多是考虑功能的可视化表达。但是更常态的是，业务有N个数据，解决了一个功能提升了1个数据，造成了其他N-1个数据的下降。在我的实例中就是商业化收益数据。

这也是之前接触业务常不会考虑的一点：支撑着这个业务/功能赚钱的逻辑是什么。这么说可能过于俗气了，但是这是决定业务生死的重要指标。所以在开始业务设计的时候，需要非常明确商业模式是什么，最好能有对应影响指标的设计经验。避免在之后的迭代中踩坑。当然还有一种情况是，体验和指标，核心指标和一般指标无法同时满足的情况。那么就需要取舍，通常是业务方一起讨论决策的。
